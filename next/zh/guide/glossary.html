术语表管理 ​

术语表用于确保专业术语在翻译过程中保持一致。每个项目拥有独立的术语表，术语规则在翻译时自动应用。

概述 ​

术语表与项目绑定，术语条目存储在项目级别。启用术语表后，翻译引擎会在每段翻译时自动查询匹配的术语，将其作为提示传给 LLM，确保术语翻译的一致性。

启用术语表 ​

  1. 进入项目工作区
  2. 在项目设置中启用 术语表 开关（glossary_enabled）
  3. 添加术语条目后，翻译时将自动应用术语规则

TIP

术语表默认关闭，需要手动启用才会在翻译中生效。

术语条目 ​

每个术语条目包含以下字段：

字段类型说明
sourcestring原始语言中的术语（必填）
targetstring翻译后的术语（必填）
case_sensitiveboolean匹配时是否区分大小写（默认 false）
notesstring术语的补充说明（可选）

示例 ​

源术语 (EN)目标术语 (ZH)区分大小写备注
APIAPI否不翻译
KubernetesKubernetes否不翻译
container容器否
deployment部署否
MySQLMySQL是数据库名称，区分大小写

创建术语条目 ​

  1. 进入项目工作区的 术语表 选项卡
  2. 点击 新建术语条目
  3. 在侧边抽屉中填写源术语、目标术语、是否区分大小写及备注
  4. 确认保存

重复检查

同一项目内不允许存在相同的源术语。系统会自动将 source 转为小写进行去重校验（唯一索引 project_id + source_key）。如果提交重复的源术语，将返回 exists 错误。

编辑术语条目 ​

编辑术语条目时，如果 目标术语（target）发生变更，API 会返回 target_changed: true 标志，提示用户可能需要同步更新已翻译的内容。

导入导出 ​

CSV 导入 ​

支持通过 CSV 文件批量导入术语。在术语表面板中点击 导入 按钮，上传 CSV 文件。

CSV 格式要求：

csv
source,target,case_sensitive,notes
API,API,false,不翻译
Kubernetes,Kubernetes,false,不翻译
container,容器,false,
deployment,部署,false,
MySQL,MySQL,true,数据库名称

导入结果将返回：

  • added：成功导入的条目数
  • skipped：跳过的条目列表，包含行号、源术语和跳过原因（duplicate、invalid_entry、short_row）

CSV 导出 ​

在术语表面板中点击 导出 按钮，将当前项目的术语表导出为 CSV 文件，方便在不同项目间共享或备份。

术语同步更新 ​

当术语条目的目标术语发生变更时，已翻译的内容不会自动更新。系统提供了术语同步功能，可以批量更新已翻译段落中的旧译文。

工作流程 ​

术语同步是一个多步骤的异步流程：

  1. 影响分析 — 系统采用两阶段匹配策略分析受影响的段落：

    • 阶段 1：原文（source_text）包含术语的源术语
    • 阶段 2：译文（target_text）包含旧的目标术语
    • 仅匹配状态为 translated、edited、approved 或 rejected 的段落
    • 支持限定特定资源范围进行分析

  2. 提交同步任务 — 确认影响范围后，提交异步同步任务。系统会创建 SyncTask 记录，返回任务 ID 和状态轮询端点。

  3. 执行替换 — 后台 Worker 分批处理（每批 100 个段落），使用安全替换算法更新译文：

    • CJK 语言直接替换
    • 拉丁语系按词边界替换，避免误替换
    • 替换后的段落状态自动设为 edited，需要人工复核

  4. 查询进度 — 前端以 500ms 间隔轮询任务状态端点，实时显示处理进度。

  5. 取消任务 — 支持取消正在执行的同步任务。pending 状态直接取消，running 状态在当前批次完成后停止。

同步任务状态 ​

状态说明
pending等待执行
running正在执行
completed执行完成
failed执行失败
cancelled已取消

超时清理

超过 24 小时未完成的同步任务会被自动标记为 failed。

自动提取术语（Bootstrap） ​

LinguaFlow 支持在翻译过程中自动从源文本中提取术语，使用独立的术语抽取提示词模板。

独立提取 ​

在翻译前单独运行术语提取阶段，使用 LLM 从源文件中识别专业术语并生成术语条目。使用独立的术语抽取提示词模板（/bootstrap-prompt-templates），提取结果以 JSON 格式返回，自动写入术语表。

在执行计划的 Bootstrap 配置中启用独立提取，或在执行计划中添加提取轮次（mode: extract）。

内联提取 ​

在翻译过程中，LLM 可以同时返回新发现的术语。系统采用 First-Wins 策略合并术语：

  • 新术语自动写入运行时术语表
  • 如果同一源术语已存在但目标不同，以表中已有版本为准（跳过新增）
  • 当 inline_conflict_strategy 设为 rewrite_local 时，会自动将本批译文中的冲突目标改写为权威版本，确保文档内一致性

配置项 ​

在执行配置（Execution Profile）中可调整以下参数：

参数说明
bootstrap.enabled是否启用术语自动提取
bootstrap.max_terms_per_1000_chars每 1000 字符最多提取的术语数
bootstrap.min_source_len源术语最小长度（默认 2 个字符）
bootstrap.inline_conflict_strategy内联冲突策略（off / rewrite_local）

术语精简（Glossary Prune） ​

随着术语表增长，可能出现冗余条目、含义冲突或用例稀少的问题。术语精简功能通过 LLM 诊断术语表质量并建议清理操作。

工作流程 ​

  1. 发起精简 — 点击术语表中的 精简 按钮，系统使用精简提示词模板（/prune-prompt-templates）分析当前术语表
  2. 查看诊断 — LLM 返回每个术语的诊断信息，包含：
    • 变更建议：keep（保留）、merge（合并到另一术语）、delete（删除）
    • 诊断原因：冗余、冲突、低质量、用例过少等
    • 目标术语：合并操作时指定的目标术语
  3. 预览确认 — 查看变更预览，确认要执行的操作
  4. 应用精简 — 确认后批量执行变更

诊断配置 ​

精简操作可通过独立的精简提示词模板（/prune-prompt-templates）自定义分析策略。

关联说明 ​

术语表与项目是一对多关系，每个术语条目通过 project_id 关联到所属项目：

  • 删除项目时，该项目下的所有术语条目会被级联删除
  • 术语表不支持跨项目共享，但可以通过 CSV 导出/导入实现

最佳实践 ​

术语管理建议

  1. 提前准备 — 在翻译前整理好术语表，尤其是领域专有名词
  2. 利用自动提取 — 首次翻译时启用 Bootstrap 功能，快速生成初始术语表
  3. 区分大小写 — 对于数据库名、品牌名等需要精确匹配的术语，启用 case_sensitive
  4. 及时同步 — 修改术语后使用同步更新功能，保持已翻译内容的一致性
  5. 添加备注 — 为容易混淆的术语添加说明，帮助 LLM 理解上下文
  6. 定期精简 — 使用术语精简功能，定期清理冗余、冲突和低质量术语条目
