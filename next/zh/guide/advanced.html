高级功能 ​

LinguaFlow 提供多种高级功能，帮助您优化翻译效果。

上下文窗口 ​

上下文窗口功能为 AI 提供翻译段落前后的上下文信息，提高翻译的连贯性。

工作原理 ​

翻译每段时，系统会自动附带：

  • 前文上下文 — 当前段落之前的 N 段
  • 后文上下文 — 当前段落之后的 N 段

系统优先使用原始文本（保护前）作为上下文，使 AI 能看到可读的内容而非占位符。当设置了字符上限时，上下文会在句子边界处截断（识别 .、!、?、。、！、？、；、;、\n 等句末符号）。

上下文段落仅用于参考，不会被翻译。系统会自动跳过空段落、纯占位符段落和装饰性分隔符。

配置 ​

在翻译配置文件中设置：

yaml
context:
  enabled: true # 是否启用上下文窗口，默认 true
  before: 1 # 取前 N 段作为上下文，默认 1，范围 0-10
  after: 1 # 取后 N 段作为上下文，默认 1，范围 0-10
  max_chars: 0 # 每个上下文段落的字符数上限，默认 0（不限制），最大 10000

内容保护规则 ​

内容保护规则用于识别和保护不需要翻译的内容，将其替换为占位符 __LF_000001__ 格式（共 14 字符，最多支持 999,999 个占位符）。

默认保护规则 ​

规则说明保护内容
code代码块围栏代码块（``` ~~~）和行内代码
link链接Markdown 链接 URL、自动链接、引用链接
placeholder占位符、{var}、%s/%d/%v、$VAR/${VAR}
xmlXML/HTML 标签HTML/XML 标签及其属性

配置 ​

yaml
protect:
  enabled: true # 是否启用内容保护，默认 true
  rules: # 启用的保护规则列表
    - code
    - link
    - placeholder
    - xml

默认启用全部 4 条规则。可根据需要选择性启用。

相邻占位符合并 ​

为减少占位符数量，系统会自动合并相邻的占位符序列。例如 __LF_000001____LF_000002__ 会被合并为单个占位符，降低对 AI 的干扰。

响应修复 ​

响应修复系统自动修复 AI 返回的翻译结果中的常见问题，采用 4 级修复链。

修复级别 ​

级别修复类型说明
L1JSON 结构修复去除 BOM/零宽字符、合并对象、补全括号、移除尾逗号、转义控制字符
L2Schema 别名将 translation/result/output/data.translations 等变体标准化为 translations
L2部分恢复部分段落缺失时返回部分结果而非失败，通过 partial_threshold 控制放弃阈值
L3占位符标准化修复大小写和下划线变体，如 __lf_000001__ → __LF_000001__
L4提示词升级将错误上下文追加到系统提示词中，进行一次重试

文本模式修复 ​

当 AI 返回纯文本而非 JSON 时，系统还会执行：

  • 去除 BOM 和零宽字符
  • 去除代码块围栏
  • 支持从 [glossary] 和 [ruby] 段落中提取结构化数据

配置 ​

yaml
repair:
  enabled: true # 是否启用响应修复，默认 true
  json_structural: true # L1: JSON 结构修复，默认 true
  schema_aliases: true # L2: Schema 别名映射，默认 true
  partial: true # L2: 部分缺失恢复，默认 true
  partial_threshold: 0.5 # 缺失率阈值 (0,1]，超过此值则放弃，默认 0.5
  placeholder_normalize: true # L3: 占位符归一化，默认 true
  prompt_upgrade: true # L4: Prompt 升级重试，默认 true

批量翻译 ​

批量翻译通过执行计划的每轮配置实现，支持将大量段落分批并发翻译。

工作原理 ​

  1. 系统根据 batch_size（段落数上限）和 max_words_per_batch（字数上限）将段落分组
  2. 每批会自动扩展上下文窗口，确保翻译连贯性
  3. 使用 Worker Pool 并发处理多个批次
  4. 字数统计：CJK 字符逐字计数，非 CJK 按空格分词计数

失败降级 ​

当某批翻译失败时，系统会自动缩小批次大小重试：

  • 缩小比例由 fallback_shrink 控制（默认 0.5，即缩小一半）
  • 缩小后的最小批次为 1 个段落
  • 无法处理的段落会传递到下一轮

配置 ​

在执行计划的每轮中配置：

yaml
batch_size: 10 # 每批段落数上限，默认 10，范围 0-10000
max_words_per_batch: 0 # 每批字数上限，默认 0（不限制），最大 100000
concurrency: 3 # 并发 Worker 数，默认 3，范围 1-100
fallback_shrink: 0.5 # 失败时批次缩小比例，默认 0.5

TIP

batch_size 和 max_words_per_batch 至少需要设置一个大于 0 的值。

多轮翻译流水线 ​

通过执行计划，可以构建模式化多轮翻译流水线。每轮可以使用不同的 AI 后端、提示词模板和执行配置。

轮次模式 ​

LinguaFlow 支持两种轮次模式：

模式说明
translate翻译轮次 — 使用后端 + 翻译提示词模板 + 执行配置进行翻译
extract提取轮次 — 使用后端 + 术语抽取提示词模板抽取术语

翻译轮次是级联回退机制：每轮只处理前一轮失败的段落。提取轮次通常在翻译前执行，自动为项目术语表补充候选术语。

工作原理 ​

多轮翻译是级联回退机制：每轮只处理前一轮失败的段落，而非对成功结果进行润色。

Round 1 (translate): 翻译全部段落
  ├─ 成功 → 写入译文，流程结束
  └─ 失败 → 传递到 Round 2
Round 2 (translate): 仅翻译 Round 1 失败的段落
  ├─ 成功 → 写入译文，流程结束
  └─ 失败 → 传递到 Round 3（如有）

典型场景 ​

  1. 模型回退 — 主模型失败时自动切换备用模型重试
  2. 策略差异 — 不同轮次使用不同的批次大小、并发数或提示词，提高成功率
  3. 容错保障 — 多轮兜底确保尽可能多的段落被翻译

执行流程 ​

  1. 收集待翻译段落（跳过空段落、纯占位符段落和装饰性分隔符）
  2. 逐轮执行：
    • 构建上下文感知的批次
    • 通过 Worker Pool 并发处理
    • 缺失的段落会在同一轮内重新分批重试一次
    • 仅失败的段落（占位符校验失败或 LLM 未返回）传递到下一轮
  3. 所有轮次后仍有未翻译的段落，会记录到文档变量 _translate_failed_indices

术语提取（Bootstrap） ​

术语提取功能在翻译前自动从源文档中抽取术语表，供翻译时参考，确保术语翻译一致性。支持两种模式：

独立提取（Pre 模式） — 翻译前独立扫描整篇文档抽取术语，使用独立的术语抽取提示词模板。所有翻译轮次都能使用抽取结果。需额外一次 LLM 调用。

内联提取（Inline 模式） — 翻译时顺带抽取术语，无需额外 LLM 调用。当多个并发 Worker 对同一术语产生不同译文时，通过冲突策略处理。

独立提取配置 ​

在执行计划中配置。使用独立的术语抽取提示词模板（/bootstrap-prompt-templates）：

yaml
bootstrap:
  enabled: false # 是否启用，默认 false
  backend_id: 1 # 术语提取使用的 AI 后端 ID
  prompt_template_id: 1 # 术语抽取提示词模板 ID
  batch_size: 20 # 每批源文段数，默认 20
  concurrency: 2 # 提取并发数，默认 2
  max_terms_per_batch: 20 # 每批最大术语数，默认 20
  min_source_len: 2 # 术语最短源文长度，默认 2

内联提取配置 ​

在执行配置中设置：

yaml
glossary:
  bootstrap:
    enabled: false # 是否启用内联自举，默认 false
    max_terms_per_1000_chars: 5 # 每千字符最大术语数
    min_source_len: 2 # 最短源术语长度
    inline_conflict_strategy: rewrite-local # 冲突策略，默认 rewrite-local

冲突策略说明：

策略说明
rewrite-local后到的 Worker 将本批译文中的冲突术语替换为权威表中的版本，保证文档内一致性（默认）
off不处理冲突，先到先得，可能导致同一术语在文档内翻译不一致

每轮配置 ​

yaml
rounds:
  - mode: translate # 轮次模式：translate（翻译）或 extract（术语提取）
    backend_id: 1 # AI 后端 ID
    prompt_template_id: 1 # 提示词模板 ID（翻译轮次用翻译模板，提取轮次用抽取模板）
    profile_id: 1 # 执行配置 ID（仅翻译轮次）
    batch_size: 10 # 批次大小
    max_words_per_batch: 0 # 字数上限
    concurrency: 3 # 并发数
    fallback_shrink: 0.5 # 失败缩小比例
    retry: # 重试配置
      max_attempts: 3
      backoff_ms: 2000
      jitter: true

验证规则 ​

  • rounds 列表不能为空
  • 每轮必须指定 mode（translate 或 extract）
  • 翻译轮次必须指定有效的 backend_id、prompt_template_id、profile_id
  • 提取轮次必须指定有效的 backend_id 和 prompt_template_id
  • concurrency 必须 ≥ 1
  • fallback_shrink 必须在 [0, 1) 范围内

Ruby 注释保护 ​

对于包含 Ruby 注释（振假名）的文本，LinguaFlow 可以保护注释结构不被破坏。

工作原理 ​

提取阶段（翻译前）：

  1. 识别 <ruby>BASE<rt>READING</rt>TRAILING</ruby> 标签
  2. 合并相邻的单字注音为词级注音（如 <ruby>呪<rt>じゅ</rt></ruby><ruby>術<rt>じゅつ</rt></ruby> → 合并为 Base:"呪術", Text:"じゅじゅつ"）
  3. 从源文本中移除 Ruby 标签，仅保留基础文本
  4. 将注音信息存储到段落元数据中

恢复阶段（翻译后）：

  1. 从 AI 响应中提取注音输出（JSON 模式为结构化字段，文本模式为 [ruby] 段落）
  2. 根据 preserve_kinds 过滤注音类型
  3. 在译文中查找基础文本并插入 <ruby> 标签
  4. 匹配失败时可触发 LLM 对齐重试

注音类型 ​

类型说明
phonetic音注（读音）
semantic义训（释义）
creative创意注音

配置 ​

在执行配置中设置：

yaml
ruby:
  enabled: false # 是否启用，默认 false
  preserve_kinds: # 保留的注音类型
    - phonetic
    - semantic
    - creative

在执行计划中可配置注音对齐重试：

yaml
ruby_retry:
  enabled: false # 是否启用注音对齐重试，默认 false
  backend_id: 0 # 对齐使用的后端 ID，0 表示使用翻译主后端

速率限制 ​

为避免触发 AI 服务的速率限制，LinguaFlow 支持为每个后端配置请求速率。

工作原理 ​

  • 使用令牌桶算法控制请求速率
  • 每个后端独立维护限速器
  • 超过限制时自动等待，不会丢弃请求

配置 ​

在后端设置中配置：

yaml
rate_limit_per_minute: 60 # 每分钟最大请求数，0 表示不限速

默认值为 0（不限速）。

重试机制 ​

翻译失败时，系统会根据错误类型采用不同的重试策略。

重试策略 ​

场景策略
429/503 速率限制等待退避时间后重试同一批次，优先使用响应中的 Retry-After 头，最小等待 5 秒
网络/超时错误缩小批次大小后重试
解析失败先尝试提示词升级（L4 修复），仍失败则缩小批次
认证错误 (401/403)不重试，段落延迟到下一轮
缺失段落在同一轮内重新分批重试一次

退避策略 ​

  • 指数退避 — 第 N 次重试的等待时间为 backoff_ms × 2^(N-1)
  • 随机抖动 — 实际等待时间增加 0-100% 的随机偏移，避免多请求同时重试
  • 429 专用 — 最小等待 5 秒，优先使用服务端返回的 Retry-After 值

配置 ​

在执行计划的每轮中配置：

yaml
retry:
  max_attempts: 3 # 最大重试次数，默认 3，范围 0-10
  backoff_ms: 2000 # 基础退避时间（毫秒），默认 2000，最小 1000
  jitter: true # 是否启用随机抖动，默认 true

开发中

钩子说明

plugins:
