# 常见问题

按「第一次卡住 → 配置 → 质量 → 格式 → 数据」排序。更完整的安装与模式说明见 [安装部署](/zh/guide/installation)、[使用模式](/zh/guide/modes)。

## 第一次使用

### 应该用二进制还是 Docker？端口是多少？

| 方式                | 模式               | 默认地址                 | 说明                 |
| ------------------- | ------------------ | ------------------------ | -------------------- |
| 预编译二进制 / 双击 | 本地模式           | `http://127.0.0.1:18080` | **个人推荐**，免登录 |
| Docker 默认         | 服务器模式（预览） | `http://localhost:8080`  | 需注册/登录          |

个人试用请优先二进制本地模式。详见 [安装部署](/zh/guide/installation) 与 [使用模式](/zh/guide/modes)。

### 双击 / 启动后浏览器没打开？

1. 手动访问 `http://127.0.0.1:18080`（若被占用，终端日志里会打印实际端口）
2. 确认进程仍在运行，终端无立即退出
3. 若使用了 `--no-browser`，需自行打开浏览器
4. 公司代理 / 安全软件可能拦截本机回环访问，可换浏览器或检查系统代理例外

### 打开界面后是空白页 / 一直转圈？

1. 硬刷新（Ctrl+F5）或换一个浏览器
2. 确认访问的是日志中的地址与端口，不要混用 `18080`（本地）与 `8080`（Docker/serve）
3. 查看启动终端是否有报错
4. 从源码开发时，需前后端都起来；预编译二进制则 UI 已嵌入，一般只需一个进程

### 第一次使用必须创建很复杂的执行计划吗？

不需要。先建 **单轮翻译**：选择你的 AI 后端 + 内置「通用提示词」+ 内置「通用策略」，批次与并发用较小值即可。术语提取、质量裁决等可在跑通后再加。见 [快速开始 · Web](/zh/guide/getting-started) 与 [核心概念](/zh/guide/concepts)。

### 执行计划是什么？为什么不能直接点翻译？

**执行计划**告诉系统：用哪个 AI 后端、哪套提示词与策略、要跑几轮（翻译 / 术语提取 / 质量裁决）。  
Web 里点「翻译」时必须选一份计划，因为服务端不会替你猜 API Key 与模型。  
最短做法：只建一轮 `translate`，绑定你的后端与内置模板。说明见 [核心概念 · 执行计划](/zh/guide/concepts#执行计划为什么重要)。

### 点「开始翻译」没反应或作业立刻失败？

按顺序排查：

1. **是否选了执行计划**，且计划里绑定的后端仍存在
2. **API Key** 是否有效、余额/配额是否够用
3. 作业详情中的 **错误信息**（401/403 多为密钥或权限；429 为限速）
4. 网络是否能访问对应 API / Base URL（公司防火墙、需代理时）
5. 首次建议 `batch_size = 1`、并发 `2`，排除批次过大导致的解析失败

仍失败时加 CLI `-v` 或查看服务日志，并到 [GitHub Issues](https://github.com/MeowSalty/LinguaFlow/issues) 搜索同类问题。

### Docker 启动后无法访问？

```bash
docker ps  # 确认容器正在运行
docker logs linguaflow  # 查看日志
```

- 端口映射应为 **8080**（例：`-p 8080:8080`）
- 浏览器访问 `http://localhost:8080`
- Docker 默认是 **服务器模式**，需要注册/登录，不是本地模式的免登录体验
- 检查本机防火墙与端口占用

### 从源码构建失败？

1. 确认 Go 版本 >= 1.21
2. 确认 Node.js 版本 >= 20
3. 确认已安装 pnpm 与 [Task](https://taskfile.dev)
4. 运行 `task backend:install` 和 `task frontend:install` 安装依赖
5. 按 [安装部署 · 从源码构建](/zh/guide/installation#从源码构建) 完整步骤构建

## AI 后端配置

### API Key 无效？

- 确认 API Key 是否正确复制（无多余空格、无缺段）
- 确认 Key 未过期、未被禁用，且账号有权限调用所选模型
- 使用自定义 **Base URL** 时，确认路径是否要求 `/v1` 等后缀
- 环境变量引用（如 `${OPENAI_API_KEY}`）在进程启动前是否已 export

### 翻译速度很慢？

- 检查到 AI 服务的网络延迟与带宽
- 换更小/更快的模型做草稿
- 适当增大批次（短段）或降低并发（避免 429 重试拖慢）
- 减少上下文窗口（`context.before` / `after`）
- 检查后端是否配置了过严的 `rate_limit_per_minute`

### 翻译质量不理想？

- 启用并维护 **术语表**
- 调整或自定义提示词模板
- 指定明确的源/目标语言，避免关键场景下 `auto` 误判
- 尝试不同模型；关键内容可用多轮计划提高成功率
- 开启质量检测，必要时加 **AI 质量裁决** 降噪后再人工审校

### 兼容网关要求 stream:true？

在 AI 后端选项中开启 **流式请求**（`stream: true`）。请求以流式发起，服务端内部累积为完整响应后再处理，对外行为不变。OpenAI / Anthropic / Google 三类后端均支持。

### 频繁 429 / 速率限制？

- 降低执行计划中的 **并发**（先试 `1`～`2`）
- 减小批次或拉长重试退避
- 在后端设置合理的 `rate_limit_per_minute`
- 确认云厂商控制台的 RPM/TPM 配额

### 源语残留误报很多？

源语残留检测偏敏感（专有名词、代码、刻意保留原文等可能被标出）。可：

1. 在执行计划中增加 **质量裁决**（`adjudicate`）轮次，对 `source_residual` 做 AI 复核（见 [流水线与原理](/zh/guide/pipeline#规则质检与-ai-质量裁决)）
2. 在段落列表中按 `source_residual` 筛选后人工确认
3. 确保项目源语言不是 `auto`（`auto` 时残留检测不生效，但指定错误语言也会影响规则）

## 文件格式

### 支持哪些格式？

Markdown、HTML、DOCX、EPUB、SRT/VTT/ASS 字幕、JSON/YAML/TOML、纯文本等。详见 [格式支持](/zh/guide/formats)。

### EPUB 翻译后排版混乱？

- 确认 EPUB 文件格式是否标准、无 DRM
- 尝试按章节分批翻译
- 检查是否有特殊格式需要保护
- 用阅读器打开导出文件核对样式

### DOCX 部分内容未翻译？

当前主要处理主文档 `word/document.xml` 中的段落。页眉页脚、批注、SmartArt、图表等可能不在翻译范围内，翻译后请用 Word 打开校对。

### 字幕文件时间码被修改？

LinguaFlow 会自动保护时间码。若发现时间码被改：

- 检查文件编码是否为 UTF-8
- 确认扩展名与真实格式一致（`.srt` / `.vtt` / `.ass`）
- 避免手工破坏时间码行后再导入

### JSON / YAML 译坏了结构？

- 确认文件本身可被标准解析器打开
- 结构化格式按**字符串叶子**翻译；数字、布尔等一般不译
- YAML/TOML **注释**可能无法保留
- 译后用原格式校验器检查；必要时减小批次并重跑失败资源

## 数据与存储

### 数据存储在哪里？

- **本地模式** — SQLite，默认在系统用户配置目录下的 `LinguaFlow`（如 Windows `%APPDATA%\LinguaFlow`）
- **服务器模式（预览）** — 默认 `./data/linguaflow.db`；可用 `--data-dir` 或环境变量指定；PostgreSQL 时库内数据在数据库中，任务/资源相关本地文件仍在数据目录下

### 如何备份数据？

复制 SQLite 数据库文件即可（路径以你的 `--data-dir` / 本地配置目录为准）：

```bash
# 示例：服务器模式默认相对路径
cp ./data/linguaflow.db ./backup/linguaflow_backup.db
```

### 如何迁移数据？

将数据库文件复制到新环境对应数据目录即可。更换机器时注意同时迁移数据目录中的资源文件（若有）。

## 性能与限制

### 支持多大的文件？

取决于 AI 上下文、可用内存与网络。大 EPUB / 长文档建议按章节或资源分批作业，并配合合理的 `batch_size` / `max_words_per_batch`。

### 批次大小设多少？要不要用字数限制？

- **内置默认**多为 `batch_size: 1`（稳妥、好排错），对当前大上下文模型来说**偏保守**
- **日常吞吐**：短段可试 `8`～`20` 段/批；普通文档可试 `4`～`10`
- **更推荐**同时设 `max_words_per_batch`（与 `batch_size` 双重约束）
- 批次过大时更常见 JSON/漏段/占位符问题

详见 [快速开始 · Web · 批次怎么设](/zh/guide/getting-started#批次怎么设) 与 [流水线与原理 · 批量与并发](/zh/guide/pipeline#批量与并发)。

### 支持多少并发翻译？

这里的「并发」是**同一文件内**同时处理的批次数（`concurrency`），不是多文件并行。

- 受 API 速率限制影响大（过高易 429）
- 单批越大，并发宜略降
- 起步可用 `2`～`4`，观察限速与错误后再调
